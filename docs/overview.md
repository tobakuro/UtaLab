# UtaLab - プロダクト概要

## 1. プロダクト概要

**UtaLab** は、任意の音声ファイルをアップロードするだけで AI がそれを解析し、カラオケ採点可能な状態に自動変換する Web アプリケーション。

JOYSOUND のような従来のカラオケ採点機能を備えつつ、「音声ファイルとして存在するものなら、どんな曲でもカラオケ採点ができる」ことを最大の差別化ポイントとする。

## 2. ターゲットユーザー

- 自分の好きな曲(マイナー曲、洋楽、同人音楽など)をカラオケで歌いたい人
- 既存カラオケサービスにない曲で採点遊びをしたい学生・音楽好き
- オリジナル曲の練習をしたい作曲者・歌ってみた勢

## 3. コアバリュー

1. **どんな音源でもカラオケになる**: 従来は業者が用意した曲しか歌えなかったカラオケを、ユーザー自身の音源で実現
2. **AI による自動解析**: ボーカル分離・メロディ抽出を自動化し、ユーザーは面倒な準備作業から解放される
3. **公平な採点**: 音程・リズム・表現力を客観的に評価

## 4. 主要機能

### 4.1 カラオケ採点機能(コア)

- ブラウザからのマイク入力をリアルタイムに解析
- 伴奏音源の再生とマイク入力の時刻同期
- お手本メロディとの比較による採点
- 採点軸
  - MVP: 音程
  - 拡張: リズム、安定感、表現力(ビブラート・しゃくり・こぶし・フォール)
- 歌詞表示(対応曲のみ、拡張機能)
- スコア履歴

### 4.2 AI 音声解析機能(差別化)

ユーザーがアップロードした任意の音源に対し、サーバー側で以下を自動実行する。

- **ボーカル/伴奏分離**: Demucs によるソース分離
- **メロディ抽出**: 分離したボーカルから CREPE でピッチ列を抽出
- **伴奏トラック生成**: ボーカル除去済みの音源をカラオケ音源として利用
- **(拡張)歌詞抽出と時間同期**: WhisperX を使用

### 4.3 ユーザー機能

- アカウント登録 / ログイン(本番版)
- アップロード楽曲の管理
- スコア履歴の可視化

## 5. 技術スタック

### 5.1 フロントエンド

| 項目 | 採用技術 | 選定理由 |
| --- | --- | --- |
| フレームワーク | Next.js (App Router) + TypeScript | SSR/SSG と動的処理の両立、チームメンバーの既存経験 |
| ランタイム | Node.js 22 (LTS) | Active LTS(2027年4月までサポート) |
| スタイリング | Tailwind CSS | カスタム UI の自由度と既製コンポーネントのバランス |
| UI コンポーネント | shadcn/ui(Radix + Lucide + Geist) | アクセシビリティ対応、情報量豊富 |
| 状態管理 | React useState / useContext → Zustand | MVP は標準機能、規模拡大時に Zustand 導入 |
| パッケージマネージャー | pnpm | インストール速度とディスク効率 |

### 5.2 音声処理(クライアントサイド)

| 項目 | 採用技術 | 選定理由 |
| --- | --- | --- |
| マイク入力 | Web Audio API (`getUserMedia`) | ブラウザ標準、追加ランタイム不要 |
| リアルタイム処理 | AudioWorklet | 専用スレッドで低レイテンシ処理 |
| ピッチ検出 | pitchy (McLeod Pitch Method) | シンプルな API、clarity 値で無音判定が容易 |
| 伴奏再生 | AudioBufferSourceNode | `<audio>` より正確な時刻同期が可能 |
| 可視化 | Canvas + AnalyserNode | 波形・ピッチラインの描画 |

### 5.3 バックエンド

| 項目 | 採用技術 | 選定理由 |
| --- | --- | --- |
| API | Next.js Route Handlers | フロントと型共有、別サーバー不要 |
| 認証(本番) | Auth.js | 自前コントロール可能、無料 |
| DB(本番) | PostgreSQL + Drizzle ORM | 型推論が素直、SQL に近い構文 |
| ジョブキュー(本番) | Inngest | 型安全、Next.js との相性が良い |
| ストレージ(本番) | Cloudflare R2 | S3 互換、エグレス無料 |

### 5.4 音声解析ワーカー

| 項目 | 採用技術 | 選定理由 |
| --- | --- | --- |
| 言語 | Python 3.11 | Demucs / CREPE の動作実績最多、情報量豊富 |
| パッケージマネージャー | uv | pip より高速、venv 相当を内包 |
| ボーカル分離 | Demucs | オープンソースで最高精度のソース分離モデル |
| メロディ抽出 | CREPE | ディープラーニングベースの高精度ピッチ検出 |
| 汎用音声処理 | librosa | スペクトログラム・特徴量抽出の標準ライブラリ |
| 歌詞抽出(拡張) | WhisperX | 時刻同期付きの文字起こし |
| 実行環境(本番) | Modal.com | サーバーレス GPU、バースト処理に最適 |

### 5.5 開発環境

| 項目 | 採用技術 | 選定理由 |

### 5.6 インフラ(本番)

| 項目 | 採用技術 | 選定理由 |

## 7. データフロー

### 7.1 楽曲アップロード & 解析

1. ユーザーが Web アプリから音声ファイルをアップロード
2. Next.js API が R2 用の署名付きアップロード URL を発行
3. ブラウザから R2 へ直接アップロード(サーバー負荷軽減)
4. アップロード完了を API に通知し、Inngest に解析ジョブを投入
5. Modal 上の Python Worker が起動
   - R2 から音源を取得
   - Demucs でボーカル / 伴奏を分離
   - CREPE でお手本メロディ(ピッチ列)を抽出
   - 結果を R2 に保存、DB にメタデータを書き込み
6. ユーザーに解析完了を通知

### 7.2 カラオケ採点

1. ユーザーが曲を選択
2. ブラウザが伴奏音源とお手本メロディ(JSON)を取得
3. `getUserMedia` でマイク入力を取得
4. AudioContext を開始、時刻を基準として同期
5. AudioWorklet 内で pitchy を使いリアルタイムピッチ検出
6. お手本メロディと比較し、フレームごとにスコア算出
7. 曲終了後、総合スコアを DB に保存、結果画面を表示

## 8. Web アプリ特有の制約と対策

| 制約 | 対策 |
| --- | --- |
| iOS Safari で AudioContext はユーザー操作後にしか開始できない | 「歌う」ボタンで明示的に開始 |
| Bluetooth イヤホン使用時に音質が落ちることがある | 有線イヤホン推奨の案内を表示 |
| `getUserMedia` のデフォルト設定がピッチ検出を阻害する | `echoCancellation: false` / `noiseSuppression: false` を明示 |
| タブ非アクティブ時の処理間引き | Page Visibility API で一時停止処理 |
| スピーカー再生時のエコーが採点を狂わせる | イヤホン装着を必須として UI で案内 |

## 9. 開発フェーズ

### Phase 0: 環境構築とサウンドプログラミング学習

- Devbox 環境構築
- Python + librosa で自分の声のピッチ可視化を試す
- Web Audio API でマイク入力 → 波形表示を試す

### Phase 1: MVP(最小動作プロトタイプ)

詳細は `mvp-roadmap.md` 参照。

- ブラウザ単体のカラオケ採点
- AI 音声解析(ローカル Python)
- アップロード → 解析 → 採点の一連フロー

### Phase 2: 永続化と認証

- PostgreSQL + Drizzle
- Auth.js による認証
- スコア履歴の DB 保存

### Phase 3: クラウド化

- Python Worker を Modal に移行
- ストレージを Cloudflare R2 に移行
- ジョブキュー化(Inngest)

### Phase 4: 表現力スコアと歌詞対応

- ビブラート / しゃくり / こぶし / フォール検出
- WhisperX による歌詞抽出・同期表示

### Phase 5: UX 改善とスケール

- PWA 化
- モバイル最適化
- パフォーマンス最適化

## 10. 運用上の注意

### 10.1 著作権

- **MVP は個人利用に限定**する。公開 URL での配布はしない
- デモ用音源は著作権フリー音源(DOVA-SYNDROME、YouTube Audio Library など)を推奨
- 将来的に公開する場合は JASRAC 契約などの整理が必要

### 10.2 プライバシー

- アップロード音源の保存期間・削除ポリシーを明確にする
- 個人の歌声データの扱いを規約で明示

## 11. 今後の検討事項

- 利用者数増加時のコスト試算(Demucs は 1 曲あたり GPU で数十秒〜数分)
- スマホ内蔵マイクでの採点精度検証
- 楽曲の公開・共有機能の是非
- オフライン対応範囲(お気に入り曲のダウンロード)
